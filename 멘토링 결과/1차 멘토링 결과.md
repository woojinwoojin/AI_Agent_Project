# 멘토링 결과 정리: RAG 구조 개선 및 DB 설계 방향

## 1. 프로젝트 개요

본 프로젝트는 **가천대학교 인공지능학과 학생을 위한 AI 학과 길잡이 Agent**를 목표로 한다.  
학생들이 학사일정, 수강신청, 졸업요건, 사회봉사, 휴학/복학, 전과/재입학 등 다양한 학사 정보를 자연어 질문으로 확인할 수 있도록 하는 RAG 기반 AI Agent MVP이다.

현재 시스템은 다음 구조를 기반으로 개발 중이다.

```text
FastAPI + Jinja2 UI
↓
LangGraph Agent Workflow
↓
PostgreSQL + pgvector 기반 RAG
↓
Upstage Embedding / Upstage LLM
↓
SSE 기반 실시간 응답 스트리밍
```

## 2. 멘토링 주요 피드백

멘토링에서는 주로 **RAG 설계와 검색 효율성**에 대한 피드백을 받았다.

### 핵심 피드백

현재 학사 정보의 범위가 넓기 때문에, 모든 정보를 하나의 RAG 저장소에 단순히 넣는 방식은 장기적으로 복잡해질 수 있다.

특히 다음 문제가 발생할 수 있다.

```text
1. RAG에 저장해야 할 정보의 범위가 너무 넓어짐
2. 문서 chunk 수가 증가하면서 검색 대상이 커짐
3. 관련 없는 문서가 검색 결과에 섞일 가능성 증가
4. 검색 속도와 검색 정확도 모두 저하될 수 있음
5. 학사 정보의 최신성 관리가 어려워질 수 있음
```

따라서 멘토님은 다음과 같은 방향을 제안했다.

> 저장할 정보를 먼저 필요한 정보 위주로 선별하고,  
> 정보를 2계층으로 분류하여 적재한 뒤,  
> 사용자의 질문이 들어오면 먼저 큰 범주로 분류하고,  
> 해당 범주 안에서만 검색하도록 구조를 개선하는 것이 좋다.

예를 들어 사용자가 졸업 관련 질문을 하면 전체 DB를 검색하는 것이 아니라, 먼저 `졸업` 영역으로 분류한 뒤 졸업 관련 정보 안에서 검색하는 방식이다.

## 3. 개선 방향 요약

멘토링 피드백을 반영하여 RAG 구조를 다음과 같이 개선한다.

```text
사용자 질문
↓
Router Node
- intent 분류
- category_l1 분류
- category_l2 추정
↓
Filtered Retriever
- category_l1 기준으로 검색 범위 제한
- pgvector 기반 후보 문서 검색
↓
Reranker
- vector score
- keyword match
- category match
- priority
- recency
를 조합하여 후보 문서 재정렬
↓
Top-k 문서만 LLM context로 전달
↓
LLM 답변 생성
↓
SSE 응답 스트리밍
```

기존 구조는 다음과 같았다.

```text
질문
↓
전체 documents에서 vector search
↓
Top-k 문서
↓
LLM 답변
```

개선 후 구조는 다음과 같다.

```text
질문
↓
질문 category 분류
↓
해당 category 내부에서 vector search
↓
후보 문서 reranking
↓
Top 3~5개 문서만 LLM에 전달
↓
답변 생성
```

## 5. RAG 정보와 정형 데이터 분리

모든 정보를 RAG에 넣는 것이 아니라, 정보의 성격에 따라 구분한다.

### RAG에 넣을 정보

문장 설명이 필요하고, 규정이나 절차를 자연어로 설명해야 하는 정보는 RAG에 넣는다.

```text
졸업요건 설명
외국어 졸업인증 설명
수강신청 유의사항
수강정정/수강포기 안내
사회봉사 교과목 이수 기준
휴학/복학 신청 절차
전과/재입학 안내
학과 사무실 및 공식 확인 안내
```

### 정형 데이터 또는 Tool로 분리할 정보

정확한 날짜, 학점, 과목, 계산이 필요한 정보는 구조화된 테이블이나 Tool로 분리하는 것이 좋다.

```text

```

즉, 최종적으로는 다음 구조를 목표로 한다.


## 7. DB 설계 확장안

기존 `documents` 테이블을 완전히 버리지 않고, 필요한 필드를 추가하여 확장한다.

```sql
ALTER TABLE documents
ADD COLUMN IF NOT EXISTS category_l1 TEXT,
ADD COLUMN IF NOT EXISTS category_l2 TEXT,
ADD COLUMN IF NOT EXISTS keywords TEXT[],
ADD COLUMN IF NOT EXISTS priority INTEGER DEFAULT 2,
ADD COLUMN IF NOT EXISTS academic_year INTEGER,
ADD COLUMN IF NOT EXISTS semester TEXT,
ADD COLUMN IF NOT EXISTS is_active BOOLEAN DEFAULT TRUE,
ADD COLUMN IF NOT EXISTS metadata JSONB;
```

검색 효율을 위해 category와 active 상태에 대한 index를 추가한다.

```sql
CREATE INDEX IF NOT EXISTS idx_documents_category
ON documents (category_l1, category_l2);

CREATE INDEX IF NOT EXISTS idx_documents_active
ON documents (is_active);
```

데이터가 많아질 경우 pgvector HNSW index를 추가한다.

```sql
CREATE INDEX IF NOT EXISTS idx_documents_embedding_hnsw
ON documents
USING hnsw (embedding vector_cosine_ops);
```

최종 `documents` 테이블은 다음과 같은 형태를 갖는다.

```text
documents
- id
- title
- content
- source
- source_page
- category_l1
- category_l2
- keywords
- priority
- academic_year
- semester
- is_active
- metadata
- embedding
```

## 8. Reranker 도입 방향

멘토링 중 `ranker`에 대한 언급이 있었다. 이는 반드시 Google Ranking API를 의미한다기보다는, RAG 문맥에서는 보통 **retriever가 가져온 후보 문서들을 다시 정렬하는 reranker**를 의미한다.

따라서 지금 단계에서는 외부 Ranking API를 바로 도입하기보다, 내부 lightweight reranker를 먼저 구현하는 방향이 적절하다.

### Reranker의 역할

```text
Vector Search
→ 관련 있어 보이는 후보 문서 10~20개 검색

Reranker
→ 실제 답변 근거로 쓸 문서 3~5개 선별
```

### 점수 계산 방식

초기 reranker는 다음 요소를 조합해 최종 점수를 계산한다.

```text
final_score =
    vector_score * 0.65
  + keyword_score * 0.15
  + category_score * 0.10
  + priority_score * 0.05
  + recency_score * 0.05
```

| 점수 | 의미 |
|---|---|
| vector_score | pgvector cosine similarity |
| keyword_score | 질문 키워드와 문서 keywords/content 매칭 정도 |
| category_score | router가 예측한 category와 문서 category 일치 여부 |
| priority_score | 핵심 문서 여부 |
| recency_score | 최신 학년도 자료 여부 |

### 예시

사용자가 다음과 같이 질문한다고 가정한다.

```text
수강신청 날짜 알려줘
```

1차 pgvector 검색 결과는 다음과 같을 수 있다.

```text
1. 수강신청 안내
2. 수강정정 안내
3. 학사일정표
4. 수강포기 안내
5. 졸업요건 안내
```

reranker는 이 후보들을 다시 정렬하여 다음과 같이 최종 문서를 선택한다.

```text
1. 2026학년도 2학기 수강신청 기간
2. 예비수강신청 및 본 수강신청 일정
3. 학사일정표 중 수강신청 항목
```

## 9. 개선 후 검색 로직

기존 검색 방식은 전체 문서 대상 vector search였다.

```sql
SELECT
    id,
    title,
    content,
    source,
    source_page,
    1 - (embedding <=> %s::vector) AS score
FROM documents
WHERE is_active = TRUE
ORDER BY embedding <=> %s::vector
LIMIT %s;
```

개선 후에는 category 필터를 먼저 적용한다.

```sql
SELECT
    id,
    title,
    content,
    source,
    source_page,
    category_l1,
    category_l2,
    keywords,
    priority,
    academic_year,
    semester,
    1 - (embedding <=> %s::vector) AS vector_score
FROM documents
WHERE is_active = TRUE
  AND category_l1 = %s
ORDER BY embedding <=> %s::vector
LIMIT %s;
```

여기서 `embedding <=> query_embedding`은 cosine distance이므로, 유사도 점수로 사용할 때는 다음처럼 변환한다.

```sql
1 - (embedding <=> query_embedding) AS vector_score
```

### Fallback 구조

category 검색 결과가 부족하거나 점수가 낮은 경우, 전체 문서 검색으로 fallback한다.

```text
1차: category_l1 안에서 검색
2차: 검색 결과 부족 시 전체 documents 검색
3차: 그래도 부족하면 “제공된 자료에서 확인되지 않습니다”라고 답변
```

## 10. LangGraph 구조 반영

LangGraph Agent State에 category 정보를 추가한다.

```python
class AgentState(TypedDict, total=False):
    messages: list
    intent: str
    category_l1: str | None
    category_l2: str | None
    retrieved_docs: list[dict]
    tool_name: str | None
    tool_args: dict | None
    tool_result: dict | None
    guardrail: bool
```

Router Node는 사용자 질문을 분석하여 다음과 같은 결과를 생성한다.

```json
{
  "intent": "rag",
  "category_l1": "graduation",
  "category_l2": "foreign_language"
}
```

RAG Node는 해당 category 정보를 사용해 검색한다.

```text
rag_node
↓
category_l1 기준 후보 검색
↓
reranker 적용
↓
retrieved_docs 반환
```

## 11. 구현 우선순위

### Step 1. 정보 리스트 작성

먼저 RAG에 넣을 정보와 정형 데이터로 분리할 정보를 표로 정리한다.

```text
title
category_l1
category_l2
source
page
content
keywords
priority
is_structured
```

예시:

| title | category_l1 | category_l2 | priority |
|---|---|---|---|
| 외국어 졸업인증 기준 | graduation | foreign_language | 1 |
| 2학기 수강신청 기간 | course | registration | 1 |
| 사회봉사 이수 기준 | social_service | requirement | 1 |
| 휴학 신청 절차 | leave_return | leave | 2 |

### Step 2. 기존 문서 chunk에 category 추가

`gachon_ai_agent_docs_cleaned.json` 또는 ingest 대상 문서에 다음 필드를 추가한다.

```json
{
  "title": "외국어 졸업인증 기준",
  "category_l1": "graduation",
  "category_l2": "foreign_language",
  "priority": 1,
  "content": "...",
  "source": "졸업.pdf",
  "source_page": "p.3",
  "keywords": ["졸업", "외국어", "졸업인증"]
}
```

### Step 3. ingest.py 수정

DB 적재 시 다음 필드를 함께 저장한다.

```text
category_l1
category_l2
keywords
priority
academic_year
semester
metadata
```

### Step 4. retrieval.py 수정

기존 전체 검색 방식에서 category 기반 검색으로 변경한다.

```python
def search(query, category_l1=None, category_l2=None, k=4, candidates=15):
    ...
```

### Step 5. reranker.py 추가

검색 후보 문서를 다시 정렬하는 lightweight reranker를 추가한다.

```text
app/services/reranker.py
```

### Step 6. router_node 수정

Router Node가 intent뿐 아니라 category도 반환하도록 수정한다.

```json
{
  "intent": "rag",
  "category_l1": "course",
  "category_l2": "registration"
}
```

## 12. 기대 효과

개선된 구조를 통해 다음 효과를 기대할 수 있다.

```text
1. 전체 문서 검색보다 검색 범위가 줄어듦
2. category 기반 검색으로 관련 없는 문서가 줄어듦
3. reranker를 통해 최종 context 품질이 향상됨
4. LLM에 전달하는 context 수를 줄여 토큰 비용을 절감할 수 있음
5. 학사 정보 확장 시에도 구조적으로 관리 가능함
6. 향후 외부 Ranking API 또는 Hybrid Search로 확장 가능함
```

## 13. 발표/보고서용 정리 문장

멘토링 피드백을 반영하여 RAG 검색 구조를 개선할 예정이다.

기존에는 모든 문서를 하나의 벡터 검색 대상으로 두었기 때문에, 문서 범위가 넓어질수록 검색 정확도와 효율이 떨어질 수 있었다. 이를 개선하기 위해 학사 정보를 `category_l1`, `category_l2`의 2계층 구조로 분류하고, 사용자 질문이 들어오면 Router가 먼저 질문 영역을 판단하도록 설계한다.

이후 해당 category 내부에서 pgvector 기반 vector search를 수행하여 후보 문서를 가져오고, vector similarity, keyword match, category match, source priority, 최신성을 조합한 lightweight reranker를 통해 최종적으로 LLM context에 들어갈 문서를 선별한다.

이를 통해 검색 범위를 줄이고, 답변 근거로 사용되는 문서의 정확도를 높이며, 향후 학사 정보 범위가 확장되더라도 안정적으로 관리할 수 있는 RAG 구조를 만들 수 있다.

```text
[Ingestion]

chunking
↓
category_l1 / category_l2 / keywords / priority 부여
↓
Upstage Embedding
↓
PostgreSQL + pgvector 저장

[Query]
사용자 질문
↓
Router Node
- intent
- category_l1
- category_l2
↓
Filtered Retriever
- category_l1 기준 검색 범위 제한
- pgvector top 10~20 후보 검색
↓
Lightweight Reranker
- vector score
- keyword score
- category score
- priority score
- recency score
↓
Top 3~5 context 선택
↓
LLM 답변 생성
↓
SSE Streaming Response
```
